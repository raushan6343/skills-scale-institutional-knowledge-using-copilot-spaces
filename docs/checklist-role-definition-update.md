# Checklist: Role Definition Update

## Purpose
This checklist ensures that new or updated role definitions in OctoAcme process documentation are complete, consistent, and properly reviewed before being merged.

## When to Use This Checklist
- Adding a new role or persona to `octoacme-roles-and-personas.md`
- Updating an existing role definition with significant changes
- Creating role-specific guidance for a new function or team structure

---

## Required Fields for Role Definitions

### ☐ Role Name
- [ ] Clear, specific title that reflects the role's primary function
- [ ] Consistent with industry-standard terminology where applicable
- [ ] Unique within the organization (no naming conflicts)

### ☐ Role Summary
- [ ] 2-3 sentence overview of the role's purpose
- [ ] Describes who the role serves and primary value delivered
- [ ] Written in clear, accessible language

### ☐ Responsibilities
- [ ] List of 4-8 key responsibilities
- [ ] Each responsibility starts with an action verb
- [ ] Specific and measurable where possible
- [ ] No overlap with responsibilities exclusively owned by other roles
- [ ] Covers day-to-day activities and strategic contributions

### ☐ Goals
- [ ] 3-5 measurable or observable goals
- [ ] Aligned with organizational objectives
- [ ] Clear success indicators
- [ ] Distinct from responsibilities (outcomes vs. activities)

### ☐ Typical Communication
- [ ] Lists key communication channels and cadences
- [ ] Identifies regular meetings or touchpoints
- [ ] Describes communication artifacts produced by the role
- [ ] Reflects both formal and informal communication patterns

### ☐ Interactions with Other Roles
- [ ] Documents relationships with at least 3-5 related roles
- [ ] Describes how collaboration occurs for each relationship
- [ ] Explains handoffs, dependencies, and coordination points
- [ ] Clarifies areas of shared responsibility
- [ ] Highlights how the role contributes to project delivery, risk management, and quality

---

## Acceptance Criteria

### Content Quality
- [ ] All required fields are completed
- [ ] Language is clear, concise, and free of jargon where possible
- [ ] Technical terms are used consistently with existing documentation
- [ ] Role definition follows the same format and structure as existing roles

### Alignment & Integration
- [ ] Role definition aligns with OctoAcme's key principles (customer-first, iterative delivery, clear ownership, data-informed decisions, psychological safety)
- [ ] Interactions with existing roles are accurately described
- [ ] No contradictions with existing process documentation
- [ ] Role fits logically within the OctoAcme project lifecycle

### Completeness
- [ ] All sections of the role template are filled out
- [ ] Interactions mention how the role supports project delivery
- [ ] Interactions mention how the role contributes to risk management
- [ ] Interactions mention how the role ensures product quality
- [ ] Examples or context provided where clarification is needed

### Accessibility
- [ ] New or changed content is reviewed for clarity by someone unfamiliar with the role
- [ ] Definition would be helpful for onboarding new team members
- [ ] Content answers: "Why does this role exist?" and "How do I work with this role?"

---

## Stakeholder Review

### Internal Review
- [ ] Reviewed by a current or former holder of the role (if available)
- [ ] Reviewed by team members who frequently interact with this role
- [ ] Reviewed by Project Manager or Process Owner
- [ ] Feedback incorporated and documented

### Cross-Functional Review
- [ ] Reviewed by at least one Developer
- [ ] Reviewed by at least one Product Manager or Product Owner
- [ ] Reviewed by at least one Project Manager
- [ ] Confirmed no conflicts with other role definitions

### Documentation Owner Approval
- [ ] Reviewed by the documentation maintainer or process owner
- [ ] Confirmed alignment with documentation standards
- [ ] Approved for merge into main documentation

---

## Post-Merge Actions

### Communication
- [ ] Update announced to relevant teams
- [ ] New role definition linked in onboarding materials (if applicable)
- [ ] Referenced in related process documents if needed

### Continuous Improvement
- [ ] Feedback mechanism established for role definition refinement
- [ ] Scheduled for review during next documentation audit cycle
- [ ] Added to role definition index or directory (if maintained)

---

## Notes
- This checklist can be adapted for role-specific guidance documents beyond `octoacme-roles-and-personas.md`
- For minor clarifications or typo fixes, a subset of this checklist may be appropriate
- Document any deviations from this checklist with rationale in the pull request description
